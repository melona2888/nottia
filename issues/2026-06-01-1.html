01 Icebreaker 선정 이유: 출처 검증

AI 이미지와 영상, 이제 "생성됐는지"를 묻는 UI가 늘어난다

이제 중요한 질문은 "잘 만들었나"만이 아닙니다. 이미지와 영상이 AI로 만들어졌는지, 카메라 원본인지, 어떤 도구로 편집됐는지를 묻는 기능이 검색과 앱 안으로 들어오고 있습니다.

검색창에 "AI가 만들었어?"

Google은 Lens, AI Mode, Circle to Search, Gemini in Chrome에서 이미지의 출처와 AI 생성 여부를 묻는 흐름을 확장합니다.

SynthID, 1,000억 건 이상

Google은 SynthID 워터마크가 1,000억 개 이상의 이미지와 동영상, 6만 년 분량의 오디오에 적용됐다고 밝혔습니다.

C2PA는 편집 이력표

Content Credentials는 원본인지, 수정됐는지, 어떤 도구가 쓰였는지를 보여주는 표준입니다. Gemini 앱에서 시작해 Search와 Chrome으로 넓어집니다.

Pixel은 촬영 순간부터 표시

Pixel 카메라는 촬영된 원본이라는 정보를 남기는 방향으로 갑니다. AI 시대에는 "만든 흔적"만큼 "진짜 원본"도 중요해집니다.

기업용 탐지 API도 등장

Google Cloud에는 AI Content Detection API가 추가됩니다. 플랫폼 운영자는 업로드된 미디어를 분류하거나 라벨링하는 데 쓸 수 있습니다.

읽는 포인트: 출처가 UX가 됨

앞으로 AI 콘텐츠 검증은 별도 전문가 작업이 아니라 검색, 브라우저, 앱에서 바로 누르는 기능에 가까워집니다.

02 Main 선정 이유: 모델 핵심 신호

Claude Opus 4.8, 빠른 응답보다 "작업 판단"을 전면에 세우다

이번 발표의 핵심은 "더 많이 아는 모델"보다 "작업 중에 스스로 의심하고 점검하는 모델"입니다. 가격은 Opus 4.7과 같게 두고, 코드와 에이전트 작업에서 판단력과 검증 흐름을 끌어올린 쪽에 가깝습니다.

Opus 4.8, 가격 그대로

새 Opus 모델이 나왔지만 일반 사용 가격은 Opus 4.7과 같습니다. 입력 100만 토큰당 5달러, 출력 100만 토큰당 25달러입니다.

거짓 자신감 4배 감소

Anthropic 평가에서 Opus 4.8은 자기가 쓴 코드의 결함을 그냥 넘길 가능성이 이전 모델보다 약 4배 낮았습니다. 모르는 부분을 표시하고, 근거가 약한 결론을 덜 밀어붙이는 쪽입니다.

AI가 일을 알아서 쪼갬

Claude Code의 dynamic workflows는 큰 코딩 작업을 작은 작업으로 나누고, 수백 개의 병렬 subagent를 돌린 뒤 결과를 다시 검증합니다. 큰 마이그레이션 같은 일을 맡기는 흐름에 맞춰진 기능입니다.

빠른 모드는 3분의 1 비용

Opus 4.8 fast mode는 이전 모델 대비 2.5배 속도를 목표로 하고, 이전 fast mode보다 비용이 3분의 1로 낮아졌습니다. 반복해서 묻고 고치는 작업의 체감 비용이 줄어듭니다.

법률 벤치마크 최고점

Anthropic 발표에 실린 초기 사용자 평가에서는 Legal Agent Benchmark 최고점을 기록했다는 언급이 나옵니다. 다만 Anthropic의 공식 외부 인용이라, 독립 벤치마크처럼 읽기보다는 전문 업무 신호로 보는 편이 안전합니다.

PDF 토큰 비용 61% 절감

Databricks는 PDF와 도표 같은 비정형 자료를 다룰 때 Opus 4.7보다 토큰 비용이 61% 낮았다고 말했습니다. 문서 분석형 업무에서 비용 효율이 좋아졌다는 신호입니다.

화면 보고 클릭하는 모델

브라우저 에이전트 평가인 Online-Mind2Web에서 84%를 기록했다는 초기 사용자 평가도 공개됐습니다. 화면을 읽고 클릭하며 끝까지 처리하는 computer-use 흐름이 핵심 비교축이 되고 있습니다.

Copilot에서도 선택 가능

GitHub Copilot Pro+, Business, Enterprise 사용자는 VS Code, JetBrains, Xcode, GitHub Mobile 등에서 Opus 4.8을 모델 선택지로 만날 수 있습니다. 다만 롤아웃은 점진 적용입니다.

02 Main 선정 이유: 검색/앱 확장

Gemini 3.5 Flash, 검색과 앱의 기본 작업 엔진으로 이동

Gemini 3.5 Flash는 "빠른 답변용 모델"이 아니라 검색, 앱, 코딩 에이전트에 들어가는 실행 엔진으로 소개됐습니다. 핵심은 속도와 장기 작업을 동시에 가져가려는 방향입니다.

검색 기본 엔진으로 이동

Google은 3.5 Flash가 Gemini 앱과 Search의 AI Mode에 글로벌로 적용된다고 설명했습니다. 모델 발표가 곧바로 검색 경험으로 이어집니다.

Flash인데 장기 작업용

Google은 Flash 계열의 속도를 유지하면서 에이전트와 코딩의 long-horizon task를 겨냥한다고 말합니다.

Antigravity가 작업대

Antigravity와 함께 쓰면 파일을 정리하고, 이름을 바꾸고, 여러 단계를 거치는 워크플로를 자동으로 처리하는 데 초점이 맞춰집니다.

AI Studio에서 UI도 빠르게

Google 데모는 3.5 Flash가 AI Studio에서 체크아웃 화면 같은 웹 UI를 빠르게 만들어내는 장면을 보여줍니다.

사용자는 "답"보다 "흐름"을 봄

검색창이 단순 답변 박스에서 파일, 영상, 탭, 대화 흐름을 함께 다루는 작업 표면으로 바뀌고 있습니다.

읽는 포인트: 빠른 에이전트 경쟁

프런티어 모델 경쟁이 이제 "똑똑한가"에서 "빠르게 여러 단계를 끝내는가"로 옮겨가고 있습니다.

02 Main 선정 이유: 멀티모달 제작

Gemini Omni Flash, 영상 생성의 초점을 "대화형 편집"으로 옮기다

Gemini Omni Flash의 핵심은 영상을 한 번에 뽑는 것이 아니라, 대화하면서 계속 고치는 방식입니다. 텍스트, 이미지, 오디오, 비디오를 입력으로 받아 영상과 오디오를 만들어냅니다.

영상에서 시작하는 Omni

Google은 Omni를 "any input"에서 만들고 편집하는 모델로 설명하지만, 출발점은 영상 생성과 편집입니다.

대화로 다시 고침

장면을 바꾸거나, 카메라 각도를 바꾸거나, 특정 요소를 지우는 식으로 이전 지시를 이어가며 편집합니다.

캐릭터와 물리 유지

Google은 캐릭터 일관성, 장면의 물리감, 앞선 지시를 기억하는 편집 흐름을 강조했습니다.

입력은 네 가지

모델 카드 기준 입력은 텍스트, 이미지, 오디오, 비디오입니다. 출력은 오디오가 포함된 고해상도 영상입니다.

앱과 Flow에서 먼저

배포 채널은 Gemini app, YouTube, Google Flow, Google Flow Music으로 안내됩니다. API 평가는 개발자/기업 롤아웃 때 공개됩니다.

한계도 명확함

복잡한 움직임, 완전한 일관성, 정확한 텍스트 렌더링은 여전히 어려운 영역이라고 모델 카드가 밝힙니다.

02 Main 선정 이유: 거버넌스 신호

OpenAI, 프런티어 모델 거버넌스를 법규 대응 문서로 공개

이번 발표는 새 기능보다 운영 기준에 가깝습니다. OpenAI는 고성능 모델을 어떻게 평가하고, 어떤 위험을 보고하며, 사고가 났을 때 어떤 절차를 둘지 공개 문서로 정리했습니다.

성능표 밖의 출시 기준

프런티어 모델은 벤치마크만으로 끝나지 않습니다. 위험 평가, 완화, 보고, 보안 관리가 출시 기준의 일부가 됩니다.

법규 대응 문서

OpenAI는 California Transparency in Frontier AI Act와 EU AI Act의 General Purpose AI Code of Practice를 직접 언급했습니다.

Preparedness가 바탕

기존 Preparedness Framework가 심각한 위험을 정의하고 운영하는 기반이고, 이번 문서는 그 일부를 규제 문맥에 맞춰 공개한 것입니다.

위험 범주는 구체적

문서는 사이버 공격, CBRN 위험, 유해 조작, 통제 상실 같은 영역의 위험 평가와 완화를 다룹니다.

외부 전문가와 업데이트

모델 보고, 사고 대응, 외부 전문가 input, 프레임워크 업데이트가 함께 들어갑니다. 한 번 만든 원칙이 아니라 계속 바뀌는 운영 문서입니다.

읽는 포인트: 도입 체크리스트

앞으로 AI를 쓰는 조직은 "무슨 모델인가"와 함께 "어떤 거버넌스를 따르는가"를 같이 봐야 합니다.

03 Dev 선정 이유: 컴퓨터 사용

Codex, Windows 앱을 보고 클릭하는 작업까지 확장

Codex 업데이트의 핵심은 "코드를 읽는 도우미"에서 "앱을 보고 조작하는 작업자"로 확장되는 흐름입니다. Windows 앱을 직접 보고, 클릭하고, 입력하며 테스트와 디버깅을 돕습니다.

Windows 앱을 직접 봄

Codex app에서 eligible users는 Windows 애플리케이션을 대상으로 보기, 클릭, 입력을 요청할 수 있습니다.

호스트는 Windows 머신

프로젝트 파일, 셸, 앱 서버, 로컬 컨텍스트는 Windows 컴퓨터에 남습니다. Codex는 그 환경 위에서 작업합니다.

폰과 Mac에서 이어보기

사용자는 iOS, Android, Mac에서 진행 상황을 확인하고, 프롬프트에 답하고, 방향을 바꿀 수 있습니다.

브라우저도 더 빨라짐

이번 릴리스에는 반응성, in-app browser 속도, 안정성, 웹 호환성 개선도 포함됐습니다.

지역 제한 있음

출시 시점 기준 Windows Computer Use는 EEA, 영국, 스위스에서는 제공되지 않습니다.

바이브 코딩 관점

03 Dev 선정 이유: API 에이전트화

Google Antigravity와 Managed Agents, API 호출을 작업 세션으로 바꾸다

04 Research 선정 이유: 평가와 안전

이번 주 논문 노트: 에이전트를 어떻게 고치고 검증할 것인가

이번 주 논문은 "더 똑똑한 에이전트"보다 "에이전트를 어떻게 평가하고 고칠 것인가"에 가깝습니다. 실패 원인 분석, 스킬 업데이트, 안전한 실행, 검색 근거 확인이 반복되는 흐름입니다.

Agentic CLEAR: 실패 위치 찾기

에이전트 행동을 시스템, trace, node 수준으로 나눠 봅니다. 최종 답만 보지 않고 어디서 실패했는지 찾는 평가 프레임워크입니다. arXiv

SkillGrad: 스킬 파일도 학습

실행 실패를 근거로 에이전트의 스킬 문서를 고칩니다. 모델을 다시 학습하지 않아도 절차 파일을 반복 개선하는 방식입니다. arXiv

LACUNA: 실행 전 안전장치

LLM이 만든 코드를 바로 실행하지 않고 타입 검사로 제한합니다. 에이전트 안전을 사후 감시가 아니라 실행 전 구조로 다루려는 논문입니다. arXiv

Show Me the Code: 검색 과정을 코드로

복잡한 RAG 추론을 자연어 생각 과정 대신 실행 가능한 Python 프로그램으로 표현합니다. 중간 계산과 검색 근거를 더 쉽게 검증하려는 접근입니다. arXiv

RAGe: 답변 품질만 보지 않기

RAG 앱의 검색, 생성, 비용, 지연, 하드웨어 제약을 함께 평가합니다. 좋은 답변과 운영 가능한 시스템은 별개라는 점을 보여줍니다. arXiv
