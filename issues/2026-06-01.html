01 Icebreaker 선정 이유: 출처 검증

AI 이미지와 영상, 이제 "생성됐는지"를 묻는 UI가 늘어난다

AI 콘텐츠의 핵심 변화는 생성 품질만이 아니라 확인 방법입니다. Google은 SynthID와 C2PA Content Credentials 검증을 Search, Gemini, Chrome, Pixel, Cloud로 확장한다고 밝혔습니다.

  • 핵심 변화: 사용자가 Lens, AI Mode, Circle to Search, Gemini in Chrome에서 이미지의 AI 생성 여부나 편집 이력을 물어볼 수 있는 방향입니다.
  • 무엇이 좋아졌나: SynthID 검증은 Gemini 앱에서 이미지, 영상, 오디오로 확장됐고 Search와 Chrome에도 순차 적용됩니다.
  • 검증 포인트: Google은 1,000억 개 이상의 이미지와 동영상, 6만 년 분량의 오디오에 SynthID 워터마크를 적용했다고 설명했습니다.
  • 읽는 포인트: AI로 만든 자료를 공유할 때 "그럴듯한가"보다 "출처와 편집 이력을 확인할 수 있는가"가 더 중요해지고 있습니다.

02 Main 선정 이유: 모델 핵심 신호

Claude Opus 4.8, 빠른 응답보다 "작업 판단"을 전면에 세우다

Anthropic은 Claude Opus 4.8을 공개하면서 이전 Opus 4.7 대비 벤치마크와 협업 품질이 개선됐고, 같은 가격으로 제공된다고 밝혔습니다. 이번 발표의 중심은 단순 답변이 아니라 코드와 에이전트 작업에서의 판단력입니다.

  • 핵심 변화: claude.ai에서는 작업에 투입할 effort를 사용자가 조절할 수 있고, Claude Code에는 대규모 문제를 다루는 dynamic workflows가 추가됐습니다.
  • 무엇이 좋아졌나: fast mode는 이전 모델 대비 2.5배 속도로 동작하며, 이전 모델의 fast mode보다 비용이 3분의 1로 낮아졌다고 설명했습니다.
  • 어디서 쓰나: Claude, Claude Code, Anthropic API와 이를 채택한 개발 도구에서 점진적으로 활용됩니다.
  • 바이브 코딩 관점: 긴 코드베이스 작업에서 모델이 모르는 부분을 표시하고, 계획을 점검하고, 테스트를 기준으로 진행하는 흐름이 더 중요해졌습니다.

02 Main 선정 이유: 검색/앱 확장

Gemini 3.5 Flash, 검색과 앱의 기본 작업 엔진으로 이동

Google은 Gemini 3.5 Flash를 에이전트와 코딩을 위한 빠른 모델로 설명하며, AI Mode in Search와 Gemini 앱에도 적용한다고 밝혔습니다. 모델 발표가 개발자 콘솔에 머무르지 않고 검색 경험으로 바로 이어진 점이 중요합니다.

02 Main 선정 이유: 거버넌스 신호

OpenAI, 프런티어 모델 거버넌스를 법규 대응 문서로 공개

OpenAI는 Frontier Governance Framework를 공개하며 안전과 보안 절차를 California Transparency in Frontier AI Act, EU AI Act의 General Purpose AI Code of Practice 같은 규제 요구와 연결해 설명했습니다.

  • 핵심 변화: 기존 Preparedness Framework를 기반으로, 위험 평가와 완화, 모델 보고, 보안 위험 관리, 사고 대응, 외부 전문가 검토를 규제 문맥에 맞춰 정리했습니다.
  • 무엇이 좋아졌나: 모델 출시 전후의 안전 논의가 내부 원칙이 아니라 공개 문서와 업데이트 가능한 절차로 제시됐습니다.
  • 어디서 쓰나: 고성능 모델을 쓰는 기업과 교육 현장에서는 "어떤 모델을 쓰나"와 함께 "어떤 안전 프레임워크를 따르나"를 확인하는 기준이 됩니다.
  • 읽는 포인트: AI 활용자는 성능표만 볼 것이 아니라 위험 분류, 제한, 사고 대응 기준까지 확인해야 하는 단계로 넘어가고 있습니다.

Codex, Windows 앱을 보고 클릭하는 작업까지 확장

OpenAI의 ChatGPT release notes는 Codex 앱에서 Windows Computer Use를 지원한다고 설명했습니다. 이제 대상 사용자는 Codex가 Windows 애플리케이션을 보고, 클릭하고, 입력하면서 테스트와 디버깅을 도울 수 있습니다.

  • 핵심 변화: Windows 머신을 프로젝트 파일, 셸, 앱 서버, 로컬 컨텍스트의 호스트로 두고 모바일이나 Mac에서 진행 상황을 이어볼 수 있습니다.

  • 어디서 쓰나: Codex app의 eligible users 대상이며, 출시 시점에는 EEA, 영국, 스위스에서 Windows Computer Use가 제공되지 않습니다.

04 Research 선정 이유: 평가와 안전

이번 주 논문 노트: 에이전트를 어떻게 고치고 검증할 것인가

이번 주 연구 노트는 모델 성능 수치보다 운영 관점의 질문을 봅니다. 에이전트가 실패한 이유를 찾고, 스킬을 업데이트하고, 실행을 안전하게 제한하고, 검색 근거를 점검하는 흐름입니다.

Agentic CLEAR: Automating Multi-Level Evaluation of LLM Agents

  • What it is: LLM 에이전트의 행동을 시스템, trace, node 수준으로 분석하는 자동 평가 프레임워크입니다.
  • Technical idea: 실행 trace 위에서 반복 실패 패턴을 찾아 텍스트 피드백과 성공률 예측으로 연결합니다.
  • Read point: 에이전트 평가는 최종 답만 보지 말고 경로와 구성요소별 실패를 봐야 합니다.
  • Source: arXiv:2605.22608

SkillGrad: Optimizing Agent Skills Like Gradient Descent

  • What it is: 에이전트 스킬 파일을 반복 실행 결과에 맞춰 개선하는 방법입니다.
  • Technical idea: 실행 실패를 loss evidence로 보고 텍스트 기반 gradient와 momentum memory로 스킬 패치를 만듭니다.
  • Read point: 에이전트의 지식은 모델 가중치가 아니라 재사용 절차 파일에서도 점진적으로 개선될 수 있습니다.
  • Source: arXiv:2605.27760

LACUNA: Safe Agents as Recursive Program Holes

  • What it is: LLM이 작성한 코드가 런타임 흐름을 바꾸더라도 타입 검사로 안전성을 제한하는 에이전트 프로그래밍 모델입니다.
  • Technical idea: 각 행동을 typed call로 만들고, 생성 코드가 주변 프로그램에 맞게 type-check된 뒤에만 실행합니다.
  • Read point: 에이전트 실행 안전성은 사후 감시뿐 아니라 실행 전 거부와 원자성으로도 설계할 수 있습니다.
  • Source: arXiv:2605.28617

Retrieval is Cheap, Show Me the Code

  • What it is: 다중 단계 RAG 추론을 자연어 사고 과정 대신 실행 가능한 Python 프로그램으로 표현하는 접근입니다.
  • Technical idea: retrieval과 QA 도구를 변수와 실행 trace로 노출해 중간 상태를 확인하고 self-repair를 가능하게 합니다.
  • Read point: 복잡한 검색형 질문은 더 많은 검색보다 검증 가능한 중간 계산 구조가 중요할 수 있습니다.
  • Source: arXiv:2605.12975

RAGe: A Retrieval-Augmented Generation Evaluation Framework

  • What it is: RAG 앱의 검색, 생성, 리소스 사용을 함께 비교해 구성요소 선택을 돕는 평가 프레임워크입니다.
  • Technical idea: chunking, vector database, embedding, retriever 조합을 하드웨어 제약과 품질 지표에 연결합니다.
  • Read point: RAG 품질은 답변 점수만이 아니라 비용, 지연, 장비 제약과 같이 봐야 합니다.
  • Source: arXiv:2605.27445
