01 Icebreaker 선정 이유: 출처 검증

AI 이미지, 이제 "누가 만들었나"를 바로 확인하는 쪽으로 간다

AI 콘텐츠가 선거, 뉴스, 쇼핑, SNS에 섞이는 시대에는 생성 여부를 따로 조사하는 것만으로 부족합니다. OpenAI는 SynthID, C2PA, 공개 검증 도구를 함께 쓰는 출처 확인 흐름을 선거 안전 문맥에서 다시 꺼냈습니다.

보이지 않는 워터마크

SynthID는 이미지 안에 보이지 않는 신호를 심는 방식입니다. 스크린샷처럼 형태가 바뀐 뒤에도 일부 확인 가능하도록 설계됐습니다.

C2PA는 메타데이터 영수증

C2PA는 이미지가 어디서 왔고 어떤 도구로 수정됐는지 기록하는 표준입니다. 워터마크보다 더 많은 정보를 줄 수 있지만, 메타데이터가 유지돼야 합니다.

검증 사이트 미리보기

OpenAI는 플랫폼 밖에서 만난 이미지도 확인할 수 있는 공개 검증 도구를 preview한다고 밝혔습니다. 생성물 확인이 앱 안 기능에서 웹 도구로 넓어집니다.

완전한 해답은 아님

출처 표시는 딥페이크 문제를 혼자 해결하지 못합니다. 그래도 추천, 라벨링, 신고, 플랫폼 정책이 참고할 수 있는 중요한 신호가 됩니다.

독자가 볼 포인트

앞으로 AI 생성물은 "잘 만들었나"보다 "확인 가능한 흔적이 남았나"가 더 중요해집니다. 신뢰도 UX가 콘텐츠 UX의 일부가 됩니다.

02 Main 선정 이유: 기업 배포 경로

OpenAI 모델과 Codex, AWS 안에서 바로 쓴다

이번 발표의 핵심은 새 모델명이 아니라 배포 위치입니다. OpenAI 프런티어 모델과 Codex가 AWS에서 일반 제공되면서, 기업은 익숙한 보안과 조달 절차 안에서 OpenAI 기능을 검토할 수 있게 됐습니다.

AWS가 배포 통로

OpenAI 모델은 Amazon Bedrock을 통해, Codex는 Bedrock 기반 개발 흐름으로 제공됩니다. 기업 입장에서는 새 공급망보다 기존 AWS 운영 모델에 붙는 쪽입니다.

보안 검토가 빨라짐

OpenAI는 보안, 컴플라이언스, 조달, 과금, 거버넌스가 도입 장벽이었다고 설명했습니다. AWS 경로는 이 장벽을 줄이는 데 초점을 둡니다.

Commercial과 GovCloud

발표에는 Commercial과 GovCloud 리전 제공이 함께 언급됐습니다. 공공, 규제 산업, 보안 민감 조직을 의식한 배포 신호입니다.

Codex는 개발 루프 안으로

Codex는 코드 작성, 리뷰, 디버깅, 현대화 작업을 AWS 환경 안에서 돕는 개발 에이전트로 소개됐습니다. 모델 API보다 작업 흐름에 가까운 배포입니다.

다음은 보안 개발

OpenAI는 Daybreak와 Codex Security도 향후 AWS 경로에서 중요해질 수 있다고 밝혔습니다. 코드 보안 검토와 위협 모델링이 다음 전선입니다.

02 Main 선정 이유: 에이전트 신뢰성

Claude Opus 4.8, 가격보다 "스스로 점검하는 태도"가 뉴스다

Claude Opus 4.8은 더 큰 숫자만 내세운 업데이트가 아닙니다. Anthropic은 같은 가격, 더 빠른 fast mode, 큰 작업을 쪼개는 dynamic workflows, 그리고 코드 결함을 덜 놓치는 정직성 평가를 함께 강조했습니다.

Opus 가격 그대로

일반 사용 가격은 Opus 4.7과 같습니다. 입력 100만 토큰당 5달러, 출력 100만 토큰당 25달러로 안내됐습니다.

결함 방치 4배 감소

Anthropic 평가에서 Opus 4.8은 자기가 쓴 코드의 결함을 그냥 넘길 가능성이 이전 모델보다 약 4배 낮았습니다.

큰 일을 작게 쪼갬

Claude Code의 dynamic workflows는 큰 작업을 계획하고, 수백 개의 병렬 subagent를 돌린 뒤 결과를 검증합니다. 코드베이스 규모 작업을 겨냥한 기능입니다.

Fast mode 비용 하락

Opus 4.8 fast mode는 2.5배 속도와 이전 fast mode 대비 3분의 1 비용을 목표로 합니다. 반복 실행과 실험 비용을 낮추는 쪽입니다.

화면 조작 평가도 전면에

초기 사용자 평가는 browser-agent와 computer-use 성능을 주요 비교축으로 언급했습니다. 코드만 쓰는 모델에서 화면 작업 모델로 관심이 옮겨갑니다.

02 Main 선정 이유: 업무 맥락 UX

Microsoft 365 Copilot, 업무 데이터로 "깊이"를 조절한다

Microsoft 365 Copilot의 새 디자인은 예쁜 화면보다 업무 맥락을 다루는 방식이 핵심입니다. 이메일, 파일, 채팅, 회의 맥락을 바탕으로 빠른 답과 깊은 추론을 오가게 만드는 UX가 전면에 나왔습니다.

Work IQ가 맥락층

Work IQ는 사용자의 업무 신호를 바탕으로 Copilot 답변의 깊이를 조절하는 intelligence layer로 설명됐습니다.

모델 선택도 업무 안으로

Microsoft는 더 깊은 추론이 필요할 때 AI 모델 선택까지 포함된다고 밝혔습니다. 모델 선택이 개발자 도구뿐 아니라 업무 앱 UI로 들어갑니다.

처음엔 짧게, 필요하면 깊게

Copilot은 먼저 읽기 쉬운 답을 주고, 사용자가 더 다듬으면 구조, 다음 행동, 추가 맥락을 펼치는 방식으로 설계됐습니다.

앱 로딩 50% 이상 개선

Microsoft는 테스트 조건에서 Copilot 앱 로딩 시간이 50% 이상 줄었다고 밝혔습니다. AI 기능도 결국 빠른 제품 경험 위에서 살아납니다.

독자가 볼 포인트

업무 AI의 경쟁은 답변 품질만이 아닙니다. 내 맥락을 언제 쓰고, 얼마나 깊게 파고들지 사용자가 통제하는 경험이 중요해집니다.

02 Main 선정 이유: 물리 AI

NVIDIA Cosmos 3, 로봇과 자율주행용 세계 모델을 하나로 묶다

Cosmos 3는 텍스트 챗봇 경쟁과 다른 축의 발표입니다. 물리 세계를 생성하고, 이해하고, 다음 행동까지 예측하는 기능을 하나의 omni-model로 묶어 로봇, 자율주행, 시뮬레이션 쪽 개발자를 겨냥했습니다.

모델 여러 개를 하나로

이전에는 세계 생성, 제어 생성, 장면 이해, 정책 생성이 나뉘어 있었습니다. Cosmos 3는 이를 하나의 모델 흐름으로 다루겠다는 발표입니다.

물리와 행동까지 입력

텍스트, 이미지, 비디오, 오디오뿐 아니라 action까지 처리하는 구조를 내세웠습니다. 픽셀 생성보다 물리적 결과 예측에 가깝습니다.

Super와 Nano 공개

Hugging Face에는 Cosmos 3 Super와 Nano가 올라왔습니다. Nano는 워크스테이션급 GPU를 의식한 효율 모델로 설명됩니다.

합성 데이터도 같이

발표에는 synthetic data generation 데이터셋과 Diffusers 통합도 포함됐습니다. 물리 AI는 모델뿐 아니라 학습 데이터 생산이 중요합니다.

읽는 포인트: AI가 몸을 얻는 방향

챗봇은 말과 문서를 다루지만 물리 AI는 공간, 움직임, 원인과 결과를 다룹니다. 다음 큰 전장은 화면 밖의 세계입니다.

03 Dev 선정 이유: 비용 운영

GitHub Copilot, 이제 사용량과 Actions 비용을 같이 본다

Copilot 뉴스의 핵심은 새 기능보다 과금 구조입니다. 6월 1일부터 모든 Copilot 플랜은 GitHub AI Credits 기반 사용량 과금으로 움직이고, Copilot code review는 Actions minutes도 함께 씁니다.

AI Credits 기준 과금

모든 Copilot 플랜은 포함 사용량을 가진 AI Credits 기반으로 계산됩니다. 포함량을 넘으면 추가 예산 설정이 필요합니다.

Code review는 두 번 계산

Copilot code review는 AI Credits와 GitHub Actions minutes를 함께 소비합니다. 자동 리뷰를 켜기 전에 저장소와 조직 비용을 같이 봐야 합니다.

사용자별 예산 통제

조직과 엔터프라이즈 관리자는 사용자 단위 예산을 설정할 수 있습니다. AI 도구 운영이 보안 설정만큼 예산 설정을 요구하게 됐습니다.

기본 runner 설정

조직 관리자는 Copilot code review에 사용할 기본 Actions runner를 정할 수 있습니다. 리뷰 에이전트도 실행 인프라를 갖는 작업자가 됩니다.

독자가 볼 포인트

AI 코딩 도구는 이제 "무료로 많이 쓰는 보조 기능"이 아닙니다. 토큰, 실행 시간, runner, 예산을 같이 관리해야 하는 운영 자원이 됐습니다.

03 Dev 선정 이유: 화면 조작

Codex, Windows 앱을 보고 클릭하는 작업자로 확장

Codex 업데이트는 개발 에이전트가 코드 파일 안에만 머무르지 않는다는 신호입니다. eligible users는 Windows 앱을 대상으로 Codex에게 보고, 클릭하고, 입력하며 테스트와 디버깅을 하도록 요청할 수 있습니다.

Windows 앱을 직접 봄

Codex app에서 Windows 애플리케이션을 대상으로 화면 확인, 클릭, 입력을 맡길 수 있습니다. UI 디버깅이 대화 밖 실제 화면으로 연결됩니다.

작업 호스트는 Windows

프로젝트 파일, 셸, 앱 서버, 로컬 컨텍스트는 Windows 컴퓨터에 남습니다. Codex는 그 환경 위에서 조작하고 확인합니다.

모바일에서 이어서 조정

사용자는 iOS, Android, Mac에서 진행 상황을 확인하고 방향을 바꿀 수 있습니다. 개발 에이전트가 책상 밖으로 이어지는 흐름입니다.

브라우저도 빨라짐

이번 릴리스에는 응답성, in-app browser 속도, 안정성, 웹 호환성 개선도 포함됐습니다. 화면 기반 검증의 품질을 높이는 기반입니다.

지역 제한 확인

출시 시점 기준 Windows Computer Use는 EEA, 영국, 스위스에서는 제공되지 않습니다. 사용 가능 지역은 먼저 확인해야 합니다.

03 Dev 선정 이유: 오픈 코드 모델

JetBrains Mellum2, 빠른 코드용 MoE 모델을 공개하다

Mellum2는 거대한 범용 모델보다 개발 워크플로의 반복 호출을 겨냥합니다. JetBrains는 12B Mixture-of-Experts 구조지만 토큰당 2.5B 파라미터만 활성화해 지연시간과 처리량을 낮춘 오픈 모델이라고 설명했습니다.

12B지만 2.5B만 활성

MoE 구조라 전체 파라미터는 12B지만 토큰당 활성 파라미터는 2.5B입니다. 작게 굴리는 작업에 맞춘 설계입니다.

코드와 텍스트 둘 다

Mellum은 원래 코드 completion 중심이었지만, Mellum2는 요약, 라우팅, RAG, sub-agent, coding feature까지 넓혔다고 설명됩니다.

Apache 2.0 공개

모델은 Apache 2.0 라이선스로 공개됐습니다. 내부 배포나 프라이빗 워크로드를 검토하는 팀에는 중요한 조건입니다.

2배 이상 빠른 추론 주장

JetBrains는 유사 크기 모델 대비 경쟁력 있는 벤치마크와 2배 이상 빠른 inference를 강조했습니다. 실제 도입 전에는 자체 워크로드 테스트가 필요합니다.

독자가 볼 포인트

에이전트 시스템은 큰 모델 한 번보다 작은 모델 여러 번을 자주 씁니다. 라우팅과 검증용 경량 모델의 가치가 커지고 있습니다.

이번 주 연구는 에이전트가 더 많은 일을 하게 된 뒤 생기는 문제를 다룹니다. 긴 검색, 모바일 조작, 스킬 증류, 영상 편집처럼 작업 표면이 넓어질수록 평가와 교정 방식이 더 중요해집니다.
